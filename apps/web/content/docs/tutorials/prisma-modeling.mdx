# Prisma Modeling

Dans ce tutoriel, nous allons voir comment **concevoir et implémenter le modèle de données** du projet AKFC avec Prisma.

Ce chapitre s'appuie directement sur le **Domain Model** présenté précédemment.

Objectifs :

- comprendre le rôle de Prisma dans l'architecture
- traduire le domain model en schéma de base de données
- générer un client Prisma typé
- comprendre comment Prisma est utilisé dans le backend

---

# Pourquoi Prisma ?

Prisma est un **ORM moderne pour Node.js et TypeScript**.

Il permet de :

- définir un schéma de base de données déclaratif
- générer un client TypeScript typé
- effectuer des requêtes sécurisées
- gérer les migrations de base de données

Dans le projet AKFC, Prisma est utilisé pour :

- modéliser les entités métier
- accéder à la base PostgreSQL
- garantir un typage fort entre backend et base de données

---

# Le fichier schema.prisma

Le cœur de Prisma est le fichier :

```
prisma/schema.prisma
```

Ce fichier contient :

- la configuration de la base de données
- les modèles
- les relations

Exemple minimal :

```prisma
generator client {
  provider = "prisma-client-js"
}

datasource db {
  provider = "postgresql"
  url      = env("DATABASE_URL")
}
```

# Traduire le Domain Model

Nous allons maintenant traduire les entités métier.

## User

```prisma
model User {
  id           String   @id @default(cuid())
  email        String   @unique
  passwordHash String
  roleId       String

  role         Role     @relation(fields: [roleId], references: [id])

  createdAt    DateTime @default(now())
  updatedAt    DateTime @updatedAt
}
```

---

## Role

```prisma
model Role {
  id          String  @id @default(cuid())
  name        String
  description String?

  users       User[]
}
```

---

## Permission

```prisma
model Permission {
  id          String  @id @default(cuid())
  name        String
  description String?
}
```

Dans un système complet, on ajouterait souvent une **table de jointure RolePermission**.

---

## Category

```prisma
model Category {
  id          String   @id @default(cuid())
  name        String
  slug        String   @unique
  description String?

  courses     Course[]
}
```

---

## Course

```prisma
model Course {
  id          String   @id @default(cuid())
  title       String
  description String?
  categoryId  String

  category    Category @relation(fields: [categoryId], references: [id])

  createdAt   DateTime @default(now())
  updatedAt   DateTime @updatedAt
}
```

---

## Post

```prisma
model Post {
  id          String   @id @default(cuid())
  title       String
  slug        String   @unique
  excerpt     String?
  content     String

  authorId    String
  author      User     @relation(fields: [authorId], references: [id])

  createdAt   DateTime @default(now())
  updatedAt   DateTime @updatedAt
  publishedAt DateTime?
}
```

# Générer Prisma Client

Une fois le schéma défini, on génère le client :

```
pnpm prisma generate
```

Cela crée un client TypeScript dans :

```
node_modules/.prisma/client
```

Ce client est **entièrement typé**.

---

# Exemple d'utilisation

Dans le backend :

```ts
const user = await prisma.user.findUnique({
  where: { email }
})
```

Types générés automatiquement :

```ts
user: User | null
```

Cela évite énormément d'erreurs.

---

# Migrations

Prisma gère aussi les migrations de base de données.

Créer une migration :

```
pnpm prisma migrate dev
```

Prisma va :

1. comparer le schéma
2. générer SQL
3. appliquer la migration

---

# Où Prisma est utilisé dans le projet

Dans l'architecture AKFC :

```
tRPC Router
      ↓
Services
      ↓
Prisma Client
      ↓
PostgreSQL
```

Le backend ne parle **jamais directement SQL**.

---

# Bonnes pratiques

Quelques bonnes pratiques importantes :

- ne jamais exposer Prisma directement au frontend
- encapsuler Prisma dans des services
- valider les entrées avec Zod
- utiliser des DTO si nécessaire
