# Project architecture

Cette page présente la structure globale du projet.

Le repository est organisé comme un **monorepo** séparant clairement :

- l’interface utilisateur
- la logique backend
- les contrats partagés
- la documentation

Cette séparation permet :

- une architecture plus claire
- un partage de types sécurisé
- une évolution indépendante des différentes couches.

Lorsqu’un utilisateur effectue une action :

1. le **client React** déclenche une requête
2. la requête est traitée par un **router tRPC**
3. le router délègue la logique à un **service métier**
4. le service interagit avec **Prisma**
5. Prisma communique avec la **base de données**

Cette organisation permet de garder :

- des routers légers
- une logique métier centralisée
- un accès aux données bien structuré.

## Organisation du monorepo

### apps/web

Application frontend basée sur Next.js.

Elle contient :

- l’interface utilisateur
- les pages de documentation
- les composants UI
- les hooks React
- les stores Zustand

C’est également ici que se trouve la documentation pédagogique du projet.

---

### packages/backend

Contient la logique serveur.

On y trouve notamment :

- les **routers tRPC**
- les **services métier**
- la **gestion des sessions**
- l’intégration **Prisma**

Le backend n’est pas directement exposé comme une API REST, mais via **tRPC**.

---

### packages/contracts

Ce package contient les éléments partagés entre client et serveur.

Principalement :

- les **schémas Zod**
- les **types TypeScript**
- les **contrats d’API**

Cela garantit que :

- le frontend et le backend utilisent exactement les mêmes types
- les validations sont centralisées.

---

### prisma

Contient le **modèle de données** et les migrations.

On y trouve :

- `schema.prisma`
- les migrations
- la configuration de la base de données

Prisma agit comme couche d’accès aux données pour les services backend.

---

## Pourquoi cette architecture ?

Cette architecture apporte plusieurs avantages :

### séparation claire des responsabilités

Chaque couche a un rôle précis.

### partage de types sécurisé

Grâce au package `contracts`, le client et le serveur partagent les mêmes schémas.

### meilleure maintenabilité

La logique métier n’est pas mélangée avec :

- l’interface utilisateur
- la couche réseau
- la base de données.
